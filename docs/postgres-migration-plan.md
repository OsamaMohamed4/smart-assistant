# خطة الهجرة إلى PostgreSQL + pgvector

> Phase 2.1 — البند الوحيد المتبقي من المرحلة الثانية. مؤجَّل عمداً لأنه
> الوحيد الذي يتطلب تجهيزاً على Railway منك قبل أي كود.

## لماذا الهجرة؟ (تذكير)

| المشكلة الحالية (SQLite) | ما يحلّه Postgres |
|---|---|
| قاعدة البيانات ملف واحد على قرص واحد — فقدانه = فقدان كل شيء | خدمة مُدارة بنسخ احتياطي مدمج من Railway |
| better-sqlite3 متزامن: كل استعلام يوقف الـ event loop | اتصالات غير متزامنة + connection pool |
| مستحيل تشغيل أكثر من نسخة من السيرفر (horizontal scaling) | عدة نسخ تشارك نفس القاعدة |
| بحث الـ RAG خطي O(N) في الذاكرة | pgvector: فهرس HNSW يبحث في ملايين المقاطع |

## المطلوب منك قبل البدء (10 دقائق)

1. من لوحة Railway → **+ New** → **Database** → **PostgreSQL**.
2. بعد الإنشاء، فعّل pgvector: افتح الخدمة → Data → نفّذ `CREATE EXTENSION IF NOT EXISTS vector;`
   (صورة Railway الرسمية تدعمها مباشرة).
3. اربط الخدمة بمشروع smart-assistant حتى يظهر `DATABASE_URL` تلقائياً في متغيرات البيئة.
4. أخبرني — وسأنفّذ الخطوات التالية كلها.

## خطة التنفيذ (من جانبي — بعد توفر DATABASE_URL)

### المرحلة أ: طبقة قاعدة بيانات مزدوجة (بدون أي تأثير على الإنتاج)
- إضافة `pg` + طبقة `db-pg.js` بنفس واجهة `sql.*` الحالية (نفس أسماء
  الدوال، نفس المعاملات) حتى لا يتغير أي route.
- سكريبت `scripts/migrate-to-pg.js`: ينشئ الجداول في Postgres (نفس
  المخطط + `vector(1536)` بدل BLOB للـ embeddings) وينسخ كل البيانات
  من SQLite ويتحقق من التطابق (عدّ الصفوف + عينات).
- التبديل عبر متغير واحد: `DB_DRIVER=sqlite|postgres` — الافتراضي يبقى
  sqlite حتى نتأكد.

### المرحلة ب: التجربة
- تشغيل محلي كامل على Postgres (smoke test + الاختبارات اليدوية).
- على Railway: تشغيل `migrate-to-pg.js` مرة (يقرأ من ملف SQLite على
  الـ volume ويكتب إلى Postgres)، ثم تفعيل `DB_DRIVER=postgres` —
  **بدون حذف ملف SQLite** (يبقى fallback جاهزاً).

### المرحلة ج: التثبيت
- أسبوع مراقبة. إذا ظهرت مشكلة: `DB_DRIVER=sqlite` يرجع كل شيء فوراً
  (مع فقد أي بيانات كتبت بعد التبديل — لذلك التبديل يتم في وقت هادئ).
- بعد الاستقرار: تحويل الـ RAG إلى pgvector (فهرس HNSW) وحذف مسار
  الـ cosine اليدوي، ثم إزالة طبقة SQLite.

### حجم العمل المتوقع
- المرحلة أ: 4-5 أيام عمل (أغلبها طبقة `sql.*` — ~70 prepared statement).
- المرحلتان ب + ج: يومان + أسبوع مراقبة.

### المخاطر وكيف نغلقها
- **اختلاف سلوك SQL** (مثل `datetime('now')` مقابل `now()`): تُحل في
  طبقة db-pg.js المعزولة، والـ smoke test يمسك الفروق.
- **فقدان بيانات أثناء النقل**: النقل يتم والخدمة تعمل على SQLite؛
  التبديل لحظي بعد التحقق من التطابق. نافذة الخطر = الفترة بين النسخ
  والتبديل (دقائق) — ننفذها في وقت هادئ ونعيد النسخ التزايدي قبل
  التبديل مباشرة.
- **رجوع طارئ**: متغير بيئة واحد.
